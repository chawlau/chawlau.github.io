白话容器基础

容器核心

  • 容器技术的核心功能，就是通过约束和修改进程的动态表现，从而为其创造出一个“边界”
  • Cgroups 技术是用来制造约束的主要手段
  • Namespace 技术则是用来修改进程视图的主要方法

容器本质

  • 进程所需要启用的一组Namespace参数。容器就只能看到当前Namespace所限定的资源、文件、设备、状态，或者配置。
  • vm是硬件级别的隔离，有虚拟硬件
  • docker守护进程的作用，响应用户的api，持有跟容器打交道的pipe,io等

容器优势

  • 敏捷和高性能
  • 不能完全隔离
  • namespace不能隔离时间

Kubernetes默认调度器

调度算法

  • 从集群所有的节点中，根据调度算法挑选出所有可以运行该 Pod 的节点；
  • 从第一步的结果中，再根据调度算法挑选一个最符合条件的节点作为最终结果。

调度流程

  • Informer Path 启动informer来监听ectd的对象变化，将新的Pod添加进调度队列
  • 对调度器缓存进行更新，尽最大可能将集群信息Cache化，从根本上提高Predicate 和 Priority的效率
  • Scheduling Path 从Scheduler Cache获取可以运行pod的宿主机列表nodes,使用Predicate获取node,然后使用priority对上述列表进行打分，选出得分最高的node作为本次的调度结果
  • 然后将Pod对象的nodeName修改为调度的Node的名字，成为Bind

Assume和真正的绑定

  • Kubernetes 的默认调度器在 Bind 阶段，只会更新 Scheduler Cache 里的 Pod 和 Node 的信息，成为Assume
  • Assume后，调度器才会创建一个 Goroutine 来异步地向 APIServer 发起更新 Pod 的请求，来真正完成 Bind 操作。

二次确认

  • 当一个新的 Pod 完成调度需要在某个节点上运行起来之前，该节点上的 kubelet 还会通过一个叫作 Admit 的操作来再次验证该 Pod 是否确实能够运行在该节点上。

Predicates

  • GeneralPredicates,负责的是最基础的调度策略
  • PodFitsHost 检查宿主机的名字是否跟Pod的spec.nodeName一致
  • PodFitsHostPorts解决端口冲突
  • PodMatchNodeSelector 检查的是，Pod 的 nodeSelector 或者 nodeAffinity 指定的节点，是否与待考察节点匹配
  • Volume相关过滤策略
  • NoDiskConflict多个 Pod 声明挂载的持久化 Volume 是否有冲突
  • MaxPDVolumeCountPredicate 检查的条件，则是一个节点上某种类型的持久化 Volume 是不是已经超过了一定数目
  • VolumeBindingPredicate 的规则。它负责检查的，是该 Pod 对应的 PV 的 nodeAffinity 字段，是否跟某个节点的标签相匹配
  • 宿主机相关的过滤规则
  • PodToleratesNodeTaints，负责检查的就是我们前面经常用到的 Node 的“污点”机制。只有当 Pod 的 Toleration 字段与 Node 的 Taint 字段能够匹配的时候，这个 Pod 才能被调度到该节点上。
  • NodeMemoryPressurePredicate，检查的是当前节点的内存是不是已经不够充足，如果是的话，那么待调度 Pod 就不能被调度到该节点上
  • Pod 相关的过滤规则
  • PodAffinityPredicate。这个规则的作用，是检查待调度 Pod 与 Node 上的已有 Pod 之间的亲密（affinity）和反亲密（anti-affinity）关系

Predicate

  • 当开始调度一个 Pod 时，Kubernetes 调度器会同时启动 16 个 Goroutine，来并发地为集群里的所有 Node 计算 Predicates，最后返回可以运行这个 Pod 的宿主机列表。

Priorities

  • LeastRequestedPriority,就是在选择空闲资源（CPU 和 Memory）最多的宿主机
  • BalancedResourceAllocation 调度完成后，所有节点里各种资源分配最均衡的那个节点，从而避免一个节点上 CPU 被大量分配、而 Memory 大量剩余的情况

调度优先级和抢占机制

  • 解决Pod调度失败时怎么解决的问题

抢占机制

  • 抢占发生的原因，一定是一个高优先级的 Pod 调度失败。这一次，我们还是称这个 Pod 为“抢占者”，称被抢占的 Pod 为“牺牲者”(victims)
  • 通过两个队列完成
  • activeQ 都是下一个调度周期需要调度的对象
  • unschedulableQ，专门用来存放调度失败的 Pod

调度器抢占操作

  • 第一步，调度器会检查牺牲者列表，清理这些 Pod 所携带的 nominatedNodeName 字段。
  • 第二步，调度器会把抢占者的 nominatedNodeName，设置为被抢占的 Node 的名字。
  • 第三步，调度器会开启一个 Goroutine，同步地删除牺牲者。

kubenetas资源管理模型

CPU和内存资源

  • Kubernetes 里 Pod 的 CPU 和内存资源，实际上还要分为 limits 和 requests 两种情况
  • 在调度的时候，kube-scheduler 只会按照 requests 的值进行计算。而在真正设置 Cgroups 限制的时候，kubelet 则会按照 limits 的值来进行设置。

三类资源分配

  • 当 Pod 里的每一个 Container 都同时设置了 requests 和 limits，并且 requests 和 limits 值相等的时候，这个 Pod 就属于 Guaranteed 类别
  • 当 Pod 不满足 Guaranteed 的条件，但至少有一个 Container 设置了 requests。那么这个 Pod 就会被划分到 Burstable 类别
  • 如果一个 Pod 既没有设置 requests，也没有设置 limits，那么它的 QoS 类别就是 BestEffort
  • QoS 划分的主要应用场景，是当宿主机资源紧张的时候，kubelet 对 Pod 进行 Eviction（即资源回收）时需要用到的。

Qos回收资源

  • BestEffort
  • Burstable
  • Guaranteed

Cpu设置

  • 首先，你的 Pod 必须是 Guaranteed 的 QoS 类型；
  • 然后，你只需要将 Pod 的 CPU 资源的 requests 和 limits 设置为同一个相等的整数值即可。
